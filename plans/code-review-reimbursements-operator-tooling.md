## Findings (ranked by severity)

### Critical

1. **`app/services/reimbursements/bacs_xlsx.rb:64-66`** — Generated BACS spreadsheet overflows the vendored EUSA template's fixed 32-row total range.
   Failure scenario: the template's `GRAND TOTAL` formula (`BREAKDOWN!B35`, `=SUM(B3:B34)`) and the Authorisation Form's total cell only cover Excel rows 3-34. Verified by generating a real file with `RubyXL::Parser` against the committed `lib/reimbursements/templates/EUSA_BACS_template.xlsx`: the 33rd expense row overwrites the `GRAND TOTAL` label/formula outright, and expenses 34-35 land outside the summed range entirely. Nothing in `batch_processor.rb`/`build_batch_job.rb`/`nightly_batch_job.rb` caps batch size, so any batch that accumulates 33+ approved-but-unbuilt expenses (plausible during a busy week) produces a BACS file EUSA acts on where the authorized total silently no longer matches the listed payments.
   Suggested fix: Dynamically add extra rows like the 32 total rows, or manually do this for the template to increase the cap to 200.  (split into multiple submissions above that), and add a regression test generating 200c+ rows against the real template asserting the GRAND TOTAL formula survives and covers every row.

2. **`app/services/reimbursements/batch_processor.rb:94-98`** — `BatchProcessor#process` unconditionally sets `result.success = true` after the draft+batch exist, silently swallowing every error from `mark_submitted`, `upload_receipts`, `notify_producers`, and `flag_batch` — none are logged or alerted.
   Failure scenario: if `mark_submitted` fails for one expense, that expense stays `Approved` even though its receipt is already in a live, sent EUSA draft. The next Build Batch run re-drafts it — risking a double payment. The operator still receives the cheerful `batch_ready` email, which hardcodes "Producer notification emails have also been sent" regardless of whether they actually were.
   Suggested fix: propagate/log per-step failures distinctly from the "batch created" success path, and don't claim notifications were sent unless they were.

58. **`app/controllers/admin/reimbursements/review_controller.rb:61-93`** (`approve`/`reject`) — Neither action checks the expense's *current* status before mutating it, unlike every other status-changing write path in this diff (`BuildBatchJob`/`NightlyBatchJob` only select `Status::APPROVED`; `ReconcileController` excludes already-reconciled/`PAID` expenses; `BatchesController#reopen` explicitly blocks `PAID` expenses).
    Failure scenario: the Reject form is unconditionally rendered on both the Pending and Approved review tabs with no server-side status re-check. A stale page (an operator viewing the Approved tab while a batch build/reconciliation concurrently advances that expense to `Submitted`/`Paid`), a double-click, or a browser back-button resubmit lets `reject` fire against an already-`Submitted`/`Paid` expense — the controller silently flips it to `Rejected` and emails the payee "your expense was rejected" for money already in flight or already paid, while `batch_id`/`payment_confirmed_date` are left untouched (a self-contradictory state no other code path accounts for). Symmetrically, `approve` against an already-`Submitted`/`Paid` expense resets it back to `Approved` with a stale `batch_id`, making it reappear in `BuildBatchJob`'s `APPROVED` selection — risking a second BACS draft for money already sent, the same double-payment class as Critical finding #2. `review_controller_test.rb` has zero tests exercising `approve`/`reject` against a non-pending expense.
    Suggested fix: guard both actions with an explicit status check (reject restricted to `pending?`/`approved?`, approve restricted to `pending?`), mirroring the invariant `ReconcileController`/`BatchesController#reopen` already enforce.

106. **`app/controllers/admin/reimbursements/batches_controller.rb#reopen`** (with `app/services/reimbursements/store.rb`'s `revert_expense_to_approved!`/`delete_batch!`) — Reopen has no way to detect that a batch's EUSA draft was already manually sent (sending is an out-of-band Outlook step this app has no visibility into by design); its only safety gate is `paid.any?`, which only trips once reconciliation has caught up — the normal state of a just-sent batch is `Submitted`-but-not-yet-`Paid` for days to weeks, not a rare edge case.
    Failure scenario: Build Batch on Monday → operator manually sends the EUSA draft in Outlook → on Wednesday, before that period's reconciliation import, the operator reopens the same batch (e.g. to fix something, or by mistake) → every linked expense reverts to Approved, the Batch row is deleted, and `draft_cleanup_flash` calls `graph.delete_message` on what may now be an already-*sent* message — since Graph's delete moves a sent message out of Sent Items too, this destroys the operator's own record that it was ever sent. The next Build Batch/nightly run then drafts a fresh, complete **second** EUSA submission for the identical expenses — a batch-wide double-payment risk, not the single-expense scenario Critical finding #2 describes.
    Suggested fix: before allowing `reopen`, verify via Graph that the stored `draft_message_id` is still an actual Outlook draft (not merely non-nil) — refuse to reopen (or require an explicit "I confirm this was never sent" acknowledgement) if it can't be confirmed still-a-draft, and never delete a Graph message from `reopen` unless that check passes.

### High

3. **`app/services/reimbursements/amount_validation.rb:28-33`** vs **`app/services/reimbursements/airtable/mapper.rb:132`** — `AmountValidation.positive_number?` parses the amount with `Kernel#Float`, but `Mapper#expense_fields` (the actual write path) parses the same string with `String#to_f` — the two disagree on hex-looking input.
   Failure scenario: `Float("0x1A")` → `26.0` (passes validation as "positive"), but `"0x1A".to_f` → `0.0`. Posting `amount=0x1A` via `ReviewController#save` or `ExpenseEditsController#update` (both pass `params[:amount].presence` straight through) sails through validation and is silently persisted to Airtable as `£0.00`. `ReviewSupport.needs_attention_reasons` only flags a zero/nil `amount_excl_vat`, never a zero `amount`, so the corrupted expense can reach Approved/batched with no warning.
   Suggested fix: use the same numeric parser in both places (both `Float()`, or both reject non-decimal strings via a strict regex first), and add a zero/negative `amount` check to `needs_attention_reasons` as a backstop.

4. **`app/jobs/reimbursements/mailbox_poll_job.rb:126`** — No idempotency key ties a created Draft expense to its source message id.
   Failure scenario: a process crash (not a raised exception) between `store.create_expense!` succeeding and `mailbox.mark_read` running leaves the message unread with the expense already created. The next poll cycle mints a duplicate expense from the same email. The header comment's "a transient error there can't mint a duplicate" claim only covers raised exceptions, not crashes/restarts.
   Suggested fix: key created expenses by source message id (or store the message id on the expense and check for an existing match before creating).

5. **`app/jobs/reimbursements/build_batch_job.rb:28`, `mailbox_poll_job.rb:24`, `nightly_batch_job.rb:28`** — None of the three `limits_concurrency` declarations pass `duration:`, so Solid Queue defaults the lock TTL to 3 minutes.
   Failure scenario: these jobs do SharePoint uploads, Graph draft creation, Gemini extraction (RubyLLM's 120s `request_timeout`) — all plausibly longer than 3 minutes combined. Once the lock expires mid-run, Solid Queue's sweep allows a concurrent second run despite the documented single-flight guarantee, defeating the concurrency guard entirely under load.
   Suggested fix: pass an explicit `duration:` on each `limits_concurrency` call sized to the job's realistic worst-case runtime.

6. **`app/controllers/admin/reimbursements/base_controller.rb:22`** / **`app/services/reimbursements/extractor.rb:52-56`** — The interactive receipt-extraction endpoint's retry cap is dead code after the RubyLLM refactor.
   Failure scenario: `Extractor#initialize` stores `@max_attempts` but never reads it. RubyLLM's global `request_timeout = 120` (new `config/initializers/ruby_llm.rb`) plus its own default `max_retries: 3` now apply uncapped to the synchronous `ExpensesController#extract` request path — contradicting the still-present comment "Interactive extraction retries less than the background poll job." A slow/degraded Gemini can now tie up a Puma worker far longer than intended on the interactive path.
   Suggested fix: either wire `@max_attempts`/a shorter timeout into the RubyLLM call for the interactive path, or delete the now-false comment and dead ivar if the longer timeout is intentionally accepted.

7. **`app/helpers/reimbursements_helper.rb:13,32,62,90,104`** — 5 of 9 public methods in this brand-new helper have zero tests: `reimbursements_modulus_badge`, `reimbursements_effective_modulus_badge`, `reimbursements_access_check_badge`, `budget_owner_names`, `reimbursements_reasons_popover`.

   Suggested fix: add cases for each modulus result, the no-bank-details path, the access-check badge's ok/fail/skip states, `budget_owner_names` with unknown owner ids, and the popover's rendered aria attributes.

28. **`app/services/reimbursements/batch_processor.rb:96`** — `flag_batch(result, batch, :producer_notifications_sent)` unconditionally writes `producer_notifications_sent: true` to the Airtable Batch record after `notify_producers`, regardless of whether any producer email actually sent.
    Failure scenario: distinct from Critical #2 (which is about the one-time `batch_ready` email's hardcoded misleading text) — this is a *persistent* Airtable flag. `batches/show.html.erb` reads it indefinitely, so an operator checking the Batch History page weeks later still sees "Producer notifications sent: yes" even for a batch where every send failed. `batch_processor_test.rb` asserts `result.producer_notifications_sent == 0` for an all-fail scenario but never checks the Airtable-persisted flag.
    Suggested fix: only set the flag true when at least one notification actually succeeded, and persist the failure count/detail alongside it.

29. **`app/jobs/reimbursements/mailbox_poll_job.rb:74`** / **`app/services/reimbursements/mailbox_client.rb:29-40`** — Inbound-email producer attribution trusts the SMTP `From` address alone; Graph's authentication-results/SPF-DKIM-DMARC headers are never checked, and there's no per-sender throttle beyond the blanket 20-messages/5-min page size.
    Failure scenario: a spoofed `From` header lets an outsider create a Draft expense under a real producer's identity with attacker-chosen content, triggering a real Gemini extraction call under that spoofed identity.
    Suggested fix: check Graph's authentication-results header (or require SPF/DKIM pass) before attributing an email to a known submitter, and add a per-sender rate limit.

30. **`app/jobs/reimbursements/ai_check_job.rb:15-25`** with **`app/services/reimbursements/store.rb:38,52,59-61,124`** — Cache-stampede: every `AiCheckJob` after the first in a batch forces an avoidable full-table Airtable relist. `update_expense!` busts the whole `EXPENSES_KEY` on every write (no in-place patch), and each job is a fresh `Store` instance with no cross-job memoization.
    Failure scenario: N unchecked-pending expenses on one Review page load enqueues N jobs; each does a necessary write-and-bust, but every job queued after the first to finish now finds `EXPENSES_KEY` already deleted by its predecessor and is forced into a live full `list_records(:expenses)` call just to look up one already-known record id — up to N-1 unnecessary full-table calls, not from any race, just normal sequential job execution (the queue config runs 3 threads). This eats into the shared ~1,000/month Airtable budget in a way the "warm-cache page view costs zero API calls" design doc doesn't account for. Distinct from #12 (a race causing duplicate writes on the *same* expense) — this is a cascading-refetch cost across *different* expenses with zero races involved.
    Suggested fix: patch the single updated record into the cached list in place on `update_expense!` rather than deleting the whole key, or have `AiCheckJob` read via a single-record fetch path instead of the full list.
